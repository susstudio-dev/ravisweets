## ADDED Requirements

### Requirement: Design tokens live in a single source
All visual tokens — color, typography scale, spacing, radius, elevation, motion duration / easing, grain — SHALL live in **one** source of truth (`apps/storefront/src/lib/theme/tokens.ts`) and SHALL be projected to: (a) Tailwind config, (b) CSS custom properties on `:root`, (c) TypeScript exports for component-level usage. No component may define its own colour or spacing literals outside the scale.

#### Scenario: No raw colour literals in components
- **WHEN** the repo is linted
- **THEN** no component file contains hardcoded `#rrggbb`, `rgb(...)`, or `hsl(...)` values; all colours reference the token layer

#### Scenario: Token change propagates everywhere
- **WHEN** a designer edits a token in `tokens.ts`
- **THEN** Tailwind utilities, CSS custom properties, and TS constants all update on the next build with no per-component edits

### Requirement: Three-layer colour model
The colour system SHALL have three layers:
1. **Raw palette** — the brand's full colour library (brand-50..900, saffron, cream, ink, semantic success/warn/danger, neutral grey scale).
2. **Semantic role tokens** — `--color-surface`, `--color-surface-elevated`, `--color-text-primary`, `--color-text-muted`, `--color-accent`, `--color-border`, `--color-ring` — mapped from the raw palette.
3. **Flavour theme tokens** — `--theme-base`, `--theme-accent`, `--theme-glow`, `--theme-ink`, `--theme-grain-opacity` — swapped per product or category to create the "magical" theme change.

#### Scenario: Components consume semantic or theme tokens, not raw
- **WHEN** a button component is authored
- **THEN** it uses semantic tokens (e.g., `bg-[var(--color-accent)]`) or theme tokens (e.g., `bg-[var(--theme-accent)]`), never raw palette values directly

### Requirement: Per-product flavour palettes
Every `Product` SHALL carry a `theme_palette` object — shape `{ base: string, accent: string, glow: string, ink: string, grainOpacity: number }` — either **auto-derived at build time** from the product's primary image, or **hand-curated** by design when the override is present. The runtime honours the hand-curated value when provided.

#### Scenario: Auto-derivation runs at build
- **WHEN** the product catalogue is built
- **THEN** every product without a hand-curated palette has a `theme_palette` generated by the palette-extraction script, stored alongside the product

#### Scenario: Hand-curated override wins
- **WHEN** a product's metadata contains `theme_palette_override`
- **THEN** that value is used and the auto-derived value is ignored

#### Scenario: Palette meets contrast floor
- **WHEN** a palette is generated or overridden
- **THEN** the derived `--theme-ink` meets WCAG AA contrast ≥ 4.5:1 against `--theme-base`; if it does not, the extraction falls back to the semantic neutral ink

### Requirement: Theme engine swaps CSS variables at route change
A central theme engine SHALL set the `--theme-*` CSS custom properties on `:root` based on route context:
- On a product page, the active product's palette is applied.
- On a category page, the category's curated palette is applied.
- On home / static pages, the brand palette is applied.
- SSR MUST emit the correct CSS variables in the initial HTML so there is **no flash** of a wrong theme.

#### Scenario: No theme flash on first paint
- **WHEN** a user navigates directly to a product page on a cold cache
- **THEN** the first paint already carries the product's flavour palette (verified via SSR HTML snapshot)

#### Scenario: Client-side navigation cross-fades
- **WHEN** a user navigates from the listing to a product page via client-side routing
- **THEN** the theme variables transition smoothly over ~250 ms (or via `View Transitions API` where supported); reduced-motion users receive an instant swap

### Requirement: Typography scale
The typography scale SHALL use **Fraunces** (display serif, variable) and **Inter** (body sans) with a fluid scale (`clamp()`-based) across six steps: `display-xl`, `display-lg`, `display-md`, `heading`, `body`, `caption`. Line heights and tracking are tuned per step and baked into tokens.

#### Scenario: Fluid display type scales correctly
- **WHEN** the viewport resizes from 360px to 1440px
- **THEN** `display-xl` scales smoothly via `clamp()` without CLS and without hitting the page's breakpoint boundaries

### Requirement: Indic accent face (Hyderabadi signature)
The system MUST include a Devanagari and/or Telugu accent face — **Tiro Devanagari** and/or **Tiro Telugu** (Telugu preferred for the Hyderabad signal) — used sparingly on the wordmark, section dividers, and pull-quotes. It MUST be subset to the glyphs actually used, loaded with `display: swap`, and add ≤ 15 KB gzipped to the critical route's font payload.

#### Scenario: Font payload respected
- **WHEN** fonts are audited on the home route
- **THEN** the total font payload including the Indic accent face is ≤ 75 KB gzipped across weights shipped

#### Scenario: Indic accent is subset to shipped glyphs
- **WHEN** the accent face is loaded
- **THEN** its woff2 payload contains only the glyphs actually referenced in shipped copy (wordmark, category labels, pull-quotes); full-script payloads are rejected by the build

### Requirement: Paisley signature motif
The design system SHALL ship a single hand-drawn **paisley** SVG mark (`paisley.svg`) as a brand primitive, consumed via a `<Paisley>` component. It is used in exactly four places:
- Section dividers between major page sections.
- As a subtle hover/focus accent on premium surfaces (product cards, hero CTA).
- As an optional garnish option per product.
- As part of the favicon and wordmark lockup.
It MUST be ≤ 4 KB inline SVG (or external < 6 KB), its stroke / fill colour MUST be theme-driven (`--theme-accent` or `--color-accent`), and its usage is limited — never decorative-heavy, never animated continuously.

#### Scenario: Paisley is theme-aware
- **WHEN** the active flavour theme changes
- **THEN** all rendered paisley marks update their stroke / fill colour to the new `--theme-accent` without re-rendering the SVG path

#### Scenario: Paisley is not over-used
- **WHEN** any single page is reviewed
- **THEN** the paisley motif appears at no more than four distinct surfaces on that page; additional uses are rejected in review

#### Scenario: Paisley respects reduced-motion
- **WHEN** reduced-motion is on
- **THEN** no paisley mark animates; it renders as a static decorative element

### Requirement: Grain / texture overlay
A grain texture SHALL be available as a composable `<Grain>` component applying a CSS background-image at configurable opacity (default 4–6%, token-driven). The asset is ≤ 12 KB. Grain MUST be disabled when `prefers-reduced-motion: reduce` OR `Save-Data: on`.

#### Scenario: Grain disables on reduced-motion
- **WHEN** the user has reduced-motion on
- **THEN** the grain overlay is not rendered on any surface

### Requirement: Elevation and radii vocabulary
Radii SHALL use a limited scale (`sm`, `md`, `lg`, `xl`, `2xl`, `pill`, `circle`). Elevation SHALL use three tiers (`flat`, `soft`, `lifted`), each with a defined box-shadow and an optional tinted ring (`--theme-glow` at low opacity) applied only when hover/focus is active.

#### Scenario: Elevation is limited to three tiers
- **WHEN** a component uses elevation
- **THEN** it uses `flat` / `soft` / `lifted` from the token scale, not arbitrary box-shadow values

### Requirement: Canonical component vocabulary
The design system SHALL expose a documented component vocabulary under `packages/ui/` (or `apps/storefront/src/components/ui/` if kept in-app): `Button`, `Chip`, `Badge`, `Card`, `ProductCard`, `Sheet`, `Dialog`, `DropdownMenu`, `Tabs`, `Accordion`, `Tooltip`, `Toast`, `SectionHeader`, `Marquee`, `MediaFrame`. Each component exposes a small, documented prop API and consumes design tokens only.

#### Scenario: Component vocabulary is the only vocabulary
- **WHEN** a new page is built
- **THEN** it composes the existing component vocabulary or proposes a new entry with rationale; it does not re-implement a one-off `div` with styled className lists that duplicate an existing component

### Requirement: Dark mode forward-compatibility
The token and theme architecture MUST NOT preclude dark mode. Raw palette values SHALL always be accessed via semantic roles so a future `[data-theme="dark"]` scope can remap roles without touching components.

#### Scenario: Dark mode is a data-attribute away
- **WHEN** a future change introduces dark mode
- **THEN** it can be enabled by defining an alternate semantic-token set inside `[data-theme="dark"]` without modifying any component
